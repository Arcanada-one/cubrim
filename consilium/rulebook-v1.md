# Cubrim — Rulebook v1 (каноническая стартовая спецификация правила)

> **Что это.** «Rulebook v1» по DoD CUBR-0003: стартовая каноническая спецификация
> правила алгоритма кубического сжатия. Это **v1-baseline**, к которому трассируется
> вся последующая работа прототипа (CUBR-0004+) и реализации (CUBR-0005+) по принципу
> «algorithm before code» — каждый шаг реализации обязан ссылаться на правило отсюда.
>
> **Чем это НЕ является.** Это **не** финальный алгоритм и **не** догма. Каждый
> v1-выбор ниже — *референсная инстанциация* (самый просто-защитимый стартовый
> вариант, который аккуратный инженер прототипирует первым), помеченная как
> `v1-default` и **опровержимая замером** по явному критерию. Победители OQ-1..OQ-5
> выбираются по замеру на корпусе (CUBR-0004+), а не по вкусу.
>
> **Источник правды по элементам:** `PRD-CUBR-0002` §4.1–4.7 (формализация),
> §6 (OQ-1..OQ-5, открытые вопросы с критериями), §7 (V-AC), §8 (риски).
> При расхождении приоритет у PRD; этот rulebook лишь фиксирует *стартовые позиции*.

## 0. Обозначения (из PRD §4.0/§4.1)

| Символ | Значение |
|--------|----------|
| `S` | исходная входная последовательность значений (вход компрессора) |
| `L` | длина входа `S` (число элементов / позиций `0..L-1`) |
| `Φ` | детерминированная биекция адресации `index/key → x = (x_1,…,x_N)` (PRD §4.1) |
| `N` | размерность куба (число осей) |
| `B` | глобальная верхняя граница длины ребра (кандидат `256`) |
| `b_k` | фактическая длина ребра по оси `k`, `0 ≤ x_k < b_k ≤ B` |
| `C` | решётка-куб размера `b_1 × … × b_N` |
| `P ⊆ C` | населённое множество (точки со значением), `|P| ≪ |C|` |
| `ρ` | плотность населённости `|P| / |C|`, `ρ ≪ 1` |
| `gap_k` | расстояние-до-следующего по оси `k` (карта расстояний, PRD §4.3) |
| `V` | домен значений (что считаем «значением», OQ-5) |
| `D` | карта расстояний — run-кодированные ряды `gap_k` |

Краеугольный инвариант корректности всего формата — **lossless byte-exact round-trip**
(`V-AC-1`, PRD §7): декомпрессор обязан восстановить `S` бит-в-бит. Компрессор,
теряющий данные, — это баг, а не trade-off (project CLAUDE.md «Round-trip correctness
is non-negotiable»).

## 1. Пять элементов — v1 референсная инстанциация

Для каждого из 5 элементов PRD ниже зафиксирован **v1-default** (что прототип
реализует ПЕРВЫМ) и явная пометка «опровержимо критерием X».

### R1 — Элемент 1: N-мерный куб с ограниченным ребром (PRD §4.1, OQ-1/OQ-3)

**Правило.** Вход `S` отображается биекцией `Φ` в координаты куба `C`, каждое ребро
`b_k ≤ B`. Референс-`Φ` для потока позиций `0..L-1` — mixed-radix разложение индекса
по основанию `B`: `i → (i mod B, (i/B) mod B, (i/B²) mod B, …)`, `N ≈ ⌈log_B(domain)⌉`.

- **v1-default:** `Φ` = mixed-radix по позиции, обход — лексикографический по
  `(x_1,…,x_N)`. Самый просто-защитимый, биективность тривиально доказуема, `Φ⁻¹` —
  обратное mixed-radix (восстановление индекса из разрядов). *Опровержимо OQ-3:
  Hilbert/Morton/домен-`Φ` выигрывают, если поднимают локальность (см. R3).*
- **v1-default `N` = 2 (baseline).** 2D — вырожденный, наименее накладный baseline,
  на котором проще всего отлаживать round-trip и инвариант, прежде чем платить за
  рост `|C|` и оси/шапку. *Опровержимо OQ-1: 3D/4D/переменное-`N` берут верх, если
  дают лучший средний ratio при сохранении round-trip и приемлемой скорости/памяти.*
- **Почему ограниченное ребро.** `b_k ≤ B` держит каждый «разряд координаты» дёшево
  кодируемым (`B = 256` → один байт-эквивалент до сжатия) и **жёстко ограничивает
  максимальный gap по оси** (см. R3-инвариант).

### R2 — Элемент 2: разреженность (PRD §4.2)

**Правило.** Хранятся **только населённые точки** `P`; их расположение кодируется
*относительно* (R3), не абсолютно. Это N-мерный относительный аналог разреженных
матричных форматов (CSR/COO, PRD §5) — без unearned novelty: новизна, если будет,
лежит в относительном gap-кодировании координат и в «сдвиге в угол» (R5), а не в
самой идее «хранить только ненулевые».

- **v1-default:** обход `P` в лексикографическом порядке координат; на каждой точке —
  её координаты через карту расстояний (R3) + значение через бит-пакинг (R5).
  *Привязан к выбору `Φ`/порядка обхода (OQ-3); опровергается вместе с R1.*

### R3 — Элемент 3: карта расстояний gap-to-next (PRD §4.3, OQ-2-раскладка)

**Правило.** По каждой оси `k`, в зафиксированном порядке обхода населённых точек,
записывается не абсолютная координата, а расстояние до следующей населённой позиции:

```
gap_k^{(j)} = (число пропущенных слотов между соседними населёнными позициями по оси k) + 1
```

**Инвариант (обязательный, из брифа):**

```
1 ≤ gap_k ≤ b_k ≤ B
```

Это одновременно (а) проверка корректности кодера и (б) верхняя граница, фиксирующая
сырую бит-ширину поля gap (`B = 256` → ≤ 8 бит на сырой gap до сжатия в R4).

- **v1-default раскладка:** `N` отдельных потоков (по одному на ось) значений `gap_k`.
  Проще всего кодировать и отлаживать поосно. *Опровержимо OQ-2: единый
  интерливленный / скан-порядок поток берёт верх, если даёт меньше бит/точку.*

#### R3.1 — Гап-инвариант и старт от sentinel −1 (ОДНОЗНАЧНОЕ декод-правило)

> **Снятие DeepSeek-флага по off-by-one.** Ниже — единственная допустимая
> интерпретация. Реализация, отклоняющаяся от неё, нарушает round-trip и **должна
> быть отвергнута на ревью**.

**Соглашение кодирования (encode).** Координата по оси отсчитывается от виртуального
sentinel `x_k^{(-1)} = −1` (позиция «перед нулевым слотом»). Для каждой населённой
точки `j` по оси `k`:

```
gap_k^{(j)} = x_k^{(j)} − x_k^{(j-1)}          где   x_k^{(-1)} ≡ −1   (sentinel)
```

Отсюда первый gap: `gap_k^{(0)} = x_k^{(0)} − (−1) = x_k^{(0)} + 1`. Отдельной ветки
для первого элемента НЕТ — sentinel −1 делает его подчиняющимся тому же правилу.

**Соглашение декодирования (decode).** Декодер стартует с `x_k = −1` и на каждом шаге:

```
x_k ← x_k + gap_k          (старт x_k = −1, до первого населённого элемента)
```

**Однозначная семантика `gap = 1` (это и есть точка, где ломаются off-by-one):**

> `gap_k = 1` означает **НОЛЬ пропущенных слотов** — населённая позиция следует
> *немедленно* за предыдущей (или, для первого элемента, населён слот `0`: `−1 + 1 = 0`).
> `gap_k = 1` — это НЕ «один пропущенный слот». Число пропущенных слотов = `gap_k − 1`.

Таблица истинности (ось, первый элемент и шаги), для устранения любой двусмысленности:

| Ситуация | пропущено слотов | `gap_k` | decode: `x_k` после `+= gap` |
|----------|------------------|---------|------------------------------|
| первый населённый = слот 0 | 0 | 1 | `−1 + 1 = 0` |
| первый населённый = слот 3 | 3 (слоты 0,1,2) | 4 | `−1 + 4 = 3` |
| соседний слот, без пропуска | 0 | 1 | `x + 1` |
| один пропущенный слот между | 1 | 2 | `x + 2` |
| два пропущенных слота | 2 | 3 | `x + 3` |

**Проверки кодера (обязательные, fail-closed):**
1. `gap_k ≥ 1` всегда (gap = 0 запрещён — две точки не могут занимать один слот по оси
   в данном порядке; gap = 0 → баг кодера).
2. `gap_k ≤ b_k` всегда (пропуск не может превышать длину ребра). Нарушение → ошибка
   кодирования, а не молчаливое усечение.

Worked-пример (PRD §4.7, 1D, `b_1 = 8`): населённые `{0,3,7}` → `D = (1, 3, 4)`;
decode: `−1+1=0 → 0+3=3 → 3+4=7` = `{0,3,7}`. Каждый `gap ≤ 8`. ✔

### R4 — Элемент 4: компактное run-кодирование карты расстояний (PRD §4.4, OQ-2-схема)

**Правило (требование PRD, не схема).** Ряды `gap_k` в кластеризованных данных
содержат длинные серии `1,1,1,…` (внутри плотного кластера) + редкие крупные скачки
(переход между кластерами). Карта `D` ДОЛЖНА компактно кодироваться рядами.

- **v1-default схема:** **чистый RLE** — каждый ряд кодируется парами
  `(значение_gap, длина_серии)`. Самый просто-защитимый, тривиально обратимый,
  даёт нулевую неоднозначность декода. *Опровержимо OQ-2: RLE+Huffman /
  delta-of-gap+RLE / Golomb-Rice (под геометрическое распределение) / ANS берут верх,
  если уменьшают **биты на населённую точку в карте** на корпусе при сохранении
  round-trip и инварианта `gap ≤ b_k`.*
- **Prior art (честно):** RLE; распределение gap, скошенное к малым, — классический
  кандидат под Golomb-Rice/энтропийное кодирование. Новизны здесь нет.

### R5 — Элемент 5: сдвиг в угол + бит-пакинг значений известной ширины (PRD §4.5, OQ-4)

**Правило.** «Сдвиг в угол» = переиндексация населённых значений к плотному
локальному коду (компактное начало координат после того, как местоположение уже
описано картой R3). Значения пакуются встык как поток битовых последовательностей
**переменной, но заранее ИЗВЕСТНОЙ ширины** — без разделителей (no-delimiter
инвариант): ширина выводится из шапки/контекста, не из маркеров конца.

- **v1-default ширина:** **явная ширина в шапке** — единая фиксированная ширина
  `W = ⌈log2(max_value + 1)⌉` на весь файл (или, как первый шаг усложнения, ширина на
  блок из явной width-table в шапке, см. R6). Это **сознательно самая простая, самая
  детерминированная и аудируемая** схема. *Опровержимо OQ-4: ширина-на-под-куб,
  целочисленные коды (Elias/Golomb) берут верх, если уменьшают биты/значение на
  корпусе при round-trip и no-delimiter.*
- **🔒 Хедж против тихого слома round-trip (PRD §8).** v1 **ЗАПРЕЩАЕТ**
  контекстно-выводимую ширину (ширину, реконструируемую декодером из карты
  расстояний). Контекстная ширина — главный источник тихого рассогласования
  encode/decode → молчаливая потеря данных. v1 берёт ТОЛЬКО **явную** ширину
  (в шапке/width-table); контекстно-выводимая ширина **откладывается** до отдельной
  замеренной итерации (OQ-4) и допускается лишь когда round-trip-property-тест её
  подтверждает на корпусе. Решение задокументировано как явное v1-правило.
- **Prior art (честно):** фиксированная-ширина бит-пакинг = bit-packing /
  frame-of-reference; целочисленные коды = Elias/Golomb; пакетная ширина = аналог
  streamvbyte/FOR. Новизны нет.

## 2. R6 — Формат файла и поля шапки для детерминированного декода (PRD §4.6)

**Раскладка файла:**

```
┌──────────┬──────────────────────┬───────────────────────────────────┐
│  Header  │ [карта расстояний D] │ [упакованный битовый поток значений] │
│          │  (run-encoded, R4)   │  (bit-packed, R5)                   │
└──────────┴──────────────────────┴───────────────────────────────────┘
```

**Поля шапки (минимум для детерминированного декода).** Декодер обязан восстановить
всё, что нужно, исключительно из шапки + потоков, без внеполосной информации:

| Поле | Тип/смысл | Обязательно |
|------|-----------|-------------|
| `magic` | сигнатура формата (фикс. байты) | да |
| `version` | версия формата (v1 = `1`) | да |
| `mode` | режим: `0 = cube` (R1–R5) / `1 = raw-store` (R7 fallback) | да |
| `N` | размерность куба (число осей) | да (mode 0) |
| `b_k[N]` | длины рёбер `b_1..b_N` (каждое `≤ B`) | да (mode 0) |
| `B` | глобальная граница ребра (для валидации `b_k ≤ B`) | да (mode 0) |
| `L` | длина входа `S` (число позиций / элементов) | да |
| `count` | `|P|` — число населённых точек | да (mode 0) |
| `map_scheme` | id схемы кодирования карты (v1: `RLE`) | да (mode 0) |
| `value_scheme` | id схемы бит-пакинга (v1: `fixed-width` / `width-table`) | да (mode 0) |
| `W` | фиксированная ширина значения в битах (если `value_scheme=fixed-width`) | усл. |
| `width_table` | таблица ширин на блок/под-куб (если `value_scheme=width-table`) | усл. |
| `value_dict` | словарь «сдвига в угол» (плотный локальный код ↔ исходное значение), если домен требует | усл. (OQ-5) |
| `traversal` | id порядка обхода (v1: `lex`) | да (mode 0) |
| `phi` | id отображения `Φ` (v1: `mixed-radix`) | да (mode 0) |
| `crc` / `len_total` | контроль целостности (рекомендуется) | реком. |

**Порядок декода (строго обратный кодированию):** прочитать шапку → если `mode=1`
вернуть raw-блок как `S` (R7) → иначе декодировать карту `D` (R4⁻¹) → восстановить
координаты населённых точек по gap-инварианту (`x_k += gap_k`, старт `−1`, R3.1) →
прочитать значения по известной ширине (R5, no-delimiter) → применить `Φ⁻¹` →
восстановить `S` бит-в-бит (`V-AC-1`).

## 3. R7 — 🔒 ОБЯЗАТЕЛЬНЫЙ fallback raw-store против blowup (PRD §8, риск worst-case)

> **Это явное v1-правило, не опция.** Известный риск (PRD §8): если населённые точки
> НЕ кластеризованы (например случайная перестановка), gap-ряды распределяются
> равномерно по `[1, B]`, RLE не сжимает ничего, и выход куба может **превысить** вход
> → формат молча раздувается. v1 ОБЯЗАН это исключить.

**Правило R7 (mandatory store-mode).** Компрессор v1 ОБЯЗАН реализовать
`mode = 1 (raw-store)`:

1. Кодер вычисляет cube-кодированный размер (R1–R6).
2. **Если** `size(cube) ≥ size(raw_input) + header_overhead` (т.е. cube-режим не даёт
   выигрыша или раздувает), кодер **переключается на `mode=1`**: пишет шапку с
   `mode=1` + `L` и складывает вход `S` как есть (raw-блок).
3. Декодер при `mode=1` возвращает raw-блок как `S` напрямую.

**Гарантия ограниченного overhead.** В режиме `raw-store` итоговый размер =
`size(S) + |header|`, где `|header|` ограничен фиксированной малой константой. Формат
**никогда не раздувается выше `size(S) + bounded_overhead`** — это инвариант v1.

- **v1-default решающее правило:** «cube тогда и только тогда, когда строго меньше
  raw + overhead, иначе raw-store». *Порог `header_overhead` и точное сравнение —
  параметры, калибруемые замером (CUBR-0004), но само наличие fallback не опровержимо:
  это hard-правило против blowup.*

## 4. R8 — Доменизация входа (PRD OQ-5)

**Правило.** Что считать «значением»/`V` — определяет, обнажит ли куб структуру или
сделает данные несжимаемо-случайными.

- **v1-default:** **байтовый поток как есть** (`V` = байты входа без предобработки).
  Самый просто-защитимый: ноль предположений о домене, round-trip тривиально честен.
  *Опровержимо OQ-5: квантование числовых / пред-токенизация / раздельные кубы по
  типам берут верх, если дают бóльшую разреженность-с-кластеризацией (низкая `ρ` +
  высокая локальность) на корпусе при round-trip.*

## 5. Трассируемость к PRD и к V-AC

| Правило | PRD-якорь | OQ / риск | V-AC |
|---------|-----------|-----------|------|
| R1 (куб + ребро + `Φ`, `N=2`) | §4.1 | OQ-1, OQ-3 | V-AC-1 |
| R2 (разреженность) | §4.2 | — (CSR/COO §5) | V-AC-1 |
| R3 (карта расстояний + инвариант) | §4.3 | OQ-2 (раскладка) | V-AC-1 |
| R3.1 (sentinel −1, gap=1 семантика) | §4.3, §4.7 | DeepSeek off-by-one флаг | V-AC-1 (round-trip) |
| R4 (run-кодирование, RLE) | §4.4 | OQ-2 (схема) | V-AC-2 (ratio) |
| R5 (сдвиг в угол + явная ширина) | §4.5 | OQ-4 | V-AC-1, V-AC-2 |
| R6 (формат + шапка) | §4.6 | — | V-AC-1, V-AC-4 |
| R7 (raw-store fallback) | §4.6, §8 риск | blowup-риск | V-AC-2 (никогда не хуже raw) |
| R8 (доменизация) | §6 OQ-5 | OQ-5 | V-AC-2 |

> **Метод разрешения для всех v1-default (мандат брифа, PRD §6):** прототип
> (CUBR-0004) → прогон на бенчмарк-корпусе → замер → лог гипотезы с результатом (даже
> отклонённой, в `hypothesis-log.md`) → выбор победителя по критерию. Все числовые
> решения — по замеру, никогда «по памяти/прозе». Round-trip (`V-AC-1`) проверяется
> первым, ratio (`V-AC-2`) — вторым.

## 6. Что v1 фиксирует жёстко (НЕ опровержимо замером)

Чтобы отделить falsifiable-defaults от hard-инвариантов корректности:

1. **Lossless byte-exact round-trip** (`V-AC-1`) — краеугольный, не trade-off.
2. **Гап-инвариант** `1 ≤ gap_k ≤ b_k ≤ B` и **sentinel −1 / `gap=1`=ноль-пропусков**
   (R3.1) — единственная допустимая семантика; off-by-one-вариант отвергается.
3. **No-delimiter** для значений (R5) — ширина из шапки/контекста, не из маркеров.
4. **raw-store fallback** (R7) — формат никогда не раздувается выше `size(S)+overhead`.
5. **Детерминированный декод из шапки** (R6) — без внеполосной информации.
6. **Секретность** — этот rulebook не покидает `consilium/`/`datarim/`.

Всё остальное (`N`, `B`, `Φ`, схема карты, схема ширины, доменизация) — v1-defaults,
опровержимые критериями OQ-1..OQ-5 на корпусе.
