## Каноническая формулировка (если X → Y на Z, потому что M)

Если заменить хранение/передачу матрицы её РЕГЕНЕРАЦИЕЙ по короткому дескриптору D
(перебор, генератор, квантовый поиск или лексикографический индекс), то total_cost
на любом реальном классе не улучшится против CAS+Cubrim-1, потому что counting/pigeonhole
ограничивает покрытие дескрипторами долей 2^|D|/2^B, колмогоровская сложность реальных
матриц эмпирически ≈ их энтропийно-сжатой цене (несжимаемый остаток 94–98%), а слепой
поиск процедуры экспоненциален и Grover даёт лишь √ той же экспоненты.

## Измеренный вердикт (2026-07-16)

Кросс-девайс корпус 13.48 ГБ (devs 12.45М / prod 1.53М / www 2.82М уникальных матриц B=4096 бит):

1. **Спектр сжимаемости:** generator-дескриптор (const/arith/xor-delta/periodic≤64/xorshift32/64/LCG/RLE)
   короче zlib-9-блока лишь у **4 136 из 16 807 141 матриц (0.025%)**; экономия точных классов —
   **900 байт на весь флот**; верхняя граница всех победителей 0.016% стора. **Сырой PRNG-класс
   (единственный, где регенерация категорически бьёт кодек): 0 из 16.8М.** 94–98% матриц — без
   генераторной структуры вовсе; их эмпирическая K-оценка = Cubrim-1/zstd-цена 339–391 Б/матрицу.
2. **Перебор (реальный замер):** 15.7М кандидатов/с/процесс; 2^32-свип = 4.6 CPU-мин НА МАТРИЦУ
   (devs-корпус целиком = 6.8 лет на 16 ядрах); 2^64 = 2 330 лет; экзафлопс-переборщик: 2^96 = 2 510 лет.
   Break-even N слепого поиска не существует.
3. **Квант:** Grover √ (оптимум BBBV) сдвигает досягаемое |D| с ~50–64 до ~100–128 бит (масштаб
   AES-128: ~2^83 гейт-операций, миллионы кубитов) — при потребных d≈2700+ бит для реальных матриц
   вывод не меняется. Pigeonhole квант не обходит (Холево: n кубитов ≤ n классических бит).
4. **Лекс-индекс (под-гипотеза оператора):** в полном пространстве индекс = B бит тождественно
   (экономия 0, перебор не нужен — биекция); на встреченном подмножестве = уже построенный CAS
   (log2 M ≈ 21–24 бита); бонус порядка существует (2.07–2.38% vs raw-список; кластеризация сверх
   uniform-бара 61–77 бит/матрицу; front-coding +4–13% vs shuffle), но дельта-код сортированных
   индексов (504 Б/матрицу) на **48% хуже** zstd-19-стора в естественном порядке (339 Б/матрицу),
   а лекс-сортировка УХУДШАЕТ реальные кодеки против discovery-порядка (zstd-19 −2.7%, Cubrim-1 −1.0%):
   файловая локальность ценнее лексикографической близости.

**NO-GO по falsification:** выигрышный класс пуст экономически (0.025% матриц / ≤единицы КБ на 13.48 ГБ),
несжимаемый остаток защищён counting-стеной, перебор экспоненциален, квант даёт √, лекс-индекс
проигрывает сжатому стору. Живая часть идеи «хранить рецепт вместо результата» уже покрыта
version-chain AH-15 (GO, фактор 64.37 в router-MVP CUBR-0054) и НЕ требует нового механизма.

## total_cost = address + metadata + residual + amortized(matrix/catalog/distribution/sync/integrity)

address: дескриптор ≥ K(M) ≈ Cubrim-1-цена для 94–98% матриц · metadata: реестр генераторов + ветка
декодера (charged, AH-22) · residual: несжимаемый остаток платится полностью · amortized: поиск
дескриптора 4.6 CPU-мин/матрицу (d=32) не амортизируется ни при каком N.

## Честный потолок / граница no-win

Регенерация выигрывает только там, где K(M) ≪ |Cubrim1(M)| И дескриптор находится дёшево:
измеренная мощность этого класса на реальном флоте — 0.025% матриц с экономией в сотни байт суммарно.
OPEN-хвост (без чисел, спекулятивно): структурированный квантовый фит (амплитудная амплификация по
иерархии генераторов) — железа и метрики не существует; НЕ основание для дизайна.
